I Asked AI to Build Me a Pokémon Game in the Browser. It Delivered a Full AAA RPG.

Source: Dev.to Open Source | Date: 2/16/2026 5:12:39 PM | Lang: EN

What started as a weekend experiment turned into one of the most fun things I've shipped this year. I've been messing around with AI-assisted game dev for a few months now, and most of the time the results are... fine. A little square that moves around. Some collision detection. A basic UI. Cool for a technical demo, but completely forgettable as a game.

But this time, I decided to stop asking for "a simple RPG" and started treating the AI like a Lead AAA Game Architect. By the time the dust settled, I had orchestrated a 6-AI relay race to build CodeMon: The Kernel Uprising—a fully playable, feature-rich browser RPG built entirely in vanilla HTML, CSS, and Three.js.

No game engine. No Unity. No Unreal. Just a single .html file that feels like a AAA production.

The Spark: The "6 AI" Challenge

The catalyst for this madness was a post by EmberNoGlow on Dev.to. They laid down a challenge to the community: "Get 6 AIs to work together to create a game in HTML. This game could be a parody... Difficulty: Medium-hard... It will either be fun or boring".

I saw that and immediately thought: "Challenge accepted."

The "AI Squad" Workflow: How I Built a Team

Instead of relying on one model to do everything (which usually leads to "context drift" and hallucinations), I assigned specific roles to six different AI models. They worked in a relay—passing the code forward as each specialized system was completed.

  1. The Architect (Claude 3.5 Sonnet): Defined the foundational GS (Game State) object. This was the "brain" that made sure the shop didn't open during a battle.
  2. The Visualist (Gemini 1.5 Pro): Handled the heavy lifting of Three.js. It optimized the 60x60 grid using Instanced Meshes to keep the draw calls low and the frame rate high.
  3. The Sound Designer (GPT-4o): Wrote the entire Procedural Sound Manager. It synthesized every "hit" and "heal" using raw Web Audio API oscillators.
  4. The Debugger (DeepSeek): Polished the O(1) collision detection. It made sure you couldn't "ghost" through the Server Towers.
  5. The UI/UX Specialist (Llama 3): Crafted the CSS "Terminal" aesthetic, the custom hit-splat animations, and the live-updating HUD.
  6. The Narrative Lead (Mistral): Scripted the dialogue for the Storyteller and the "Kernel Master" boss encounter lore.

Technical Deep Dive: What’s Actually Under the Hood

1. Central Game State — Killing Mode Spaghetti

The biggest mistake in browser gamedev is scattered global variables. We solved this by creating a single, immutable source of truth: the GS object.

const GS = { mode: 'OVERWORLD', // OVERWORLD | BATTLE | SHOP | DIALOGUE player: { hp: 100, bits: 0, level: 5, questGoal: 500, // }, world: { collisionSet: new Set() } //
};

Enter fullscreen mode Exit fullscreen mode

By making every system check GS.mode first, we ensured that a random encounter couldn't trigger while the player was mid-conversation with an NPC.

2. Zero-Footprint Audio Manager

One of the most impressive parts of the build is that there is not a single audio file in the project. The Sound Designer AI built a "Procedural Sound Manager" that synthesizes tones live.

  • Walking: A soft, low-frequency square wave triggered on every step.
  • Critical Hits: A multi-frequency sawtooth burst that gives the combat "crunch".
  • The Boss Motif: A low-frequency sawtooth motif that signals high stakes.

3. "Game Juice" and Action Cameras

To move away from "boring" combat, we implemented a dynamic camera system.

  • Action Cam: During the enemy’s turn, the camera lerps to a dramatic side angle to highlight the "Syntax Error" attack.
  • Companion Drone: You aren't fighting alone. A 3D bot floats beside you and fires actual LineSegments laser geometry whenever you attack.
  • Hit Splats: When you land a hit, the DOM generates a floating number that fades out—yellow for crits, green for heals.

The Economy: Bits and Buffer Overflows

The game features a fully realized economy. You farm Bits by defeating NullPointers and Bugs in the "Dead Logic Woods." You then return to the Merchant Terminal to spend them on permanent upgrades:

Lessons Learned from the "One-File" Challenge

  1. Instanced Meshes are Non-Negotiable: Rendering 3,600 individual floor tiles will tank your FPS. Instancing them collapsed that into just 4 draw calls per biome.
  2. O(1) Collision is King: Instead of raycasting every frame, we registered every "solid" coordinate into a Set. Collision checking is now a simple .has() check, which is lightning-fast.
  3. Prompt Specificity is the Whole Game: The first version of this was a player square. This version is 1,700+ lines of structured code because I asked for architecture, not just features.

What's Next for CodeMon?

The journey doesn't end here. Now that the engine is solid, I’m planning:

  • The "Memory Leak" Swamp: A biome where movement is slowed and the UI starts to visually "corrupt."
  • Local Storage Save System: So you can keep your Level 10 Coder and your hoard of Bits across sessions.
  • Inventory UI: Moving from instant upgrades to a "backpack" of items you can use in battle.
