Why I Built a File Converter That Doesn't Upload Your Files

Source: Dev.to Open Source | Date: 2/18/2026 7:12:25 AM | Lien | Lang: EN

Every file converter website works the same way: you upload your file, wait, download the result. Your data travels to someone else's server, gets processed, and comes back.

But what if your file never had to leave your device?

The Problem with Cloud-Based Converters

I've used dozens of file conversion tools over the years. They all share the same architecture: your file goes up to a server, gets converted, and comes back down. This model has three fundamental problems.

Privacy: Your file exists on someone else's server, even if temporarily. Financial documents, medical records, personal photos — they all pass through infrastructure you don't control. Most services say they delete files after conversion, but you're trusting their word.

Speed: Upload time + processing time + download time. For a 50MB video file on a slow connection, that's minutes of waiting. The actual conversion might take 5 seconds, but the network overhead dominates.

Availability: No internet? No conversion. Server down? No conversion. Rate limited? No conversion. Your ability to convert files depends entirely on someone else's infrastructure.

Client-Side: A Different Architecture

When I built OmniConvert, I chose a radically different approach: everything runs in your browser.

Your file never leaves your device. The conversion happens locally using WebAssembly (WASM) and browser APIs. The server delivers the code; your machine does the work.

This means:

  • Zero data exposure — your files stay on your device
  • No upload wait — conversion starts instantly
  • Works offline — once loaded, no internet needed
  • No file size limits — limited only by your device's memory

How It Actually Works

The technical stack is surprisingly elegant:

  1. FFmpeg.wasm handles video and audio conversions — the same FFmpeg that powers most media processing, compiled to WebAssembly
  2. Canvas API handles image format conversions natively in the browser
  3. PDF.js + jsPDF handle document conversions
  4. SheetJS handles spreadsheet conversions

All of these run entirely in the browser tab. When you drop a file onto OmniConvert, it never touches a network request. The conversion happens in a Web Worker so your browser stays responsive.

The Trade-offs

Client-side isn't perfect. Heavy video transcoding is slower than a beefy server. The initial load downloads ~25MB of WASM binaries. And some conversions that need server-side tools (like OCR or advanced PDF manipulation) aren't possible yet.

But for 90% of everyday file conversions — images, documents, audio, video format changes — client-side is faster, more private, and more reliable than any cloud service.
