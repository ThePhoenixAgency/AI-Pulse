It feels so simple, so... Go-like. You fire off a goroutine and move on. But what happens when processImportantTask panics? Or the server restarts during a deployment? Or a downstream API it calls is temporarily down?

Your job might be lost forever, or a panic could crash your entire application. The simple "fire-and-forget" goroutine is powerful, but it's not production-safe for tasks you can't afford to lose.

This is a lesson many of us learn the hard way. That's why I built CSJD, a reliability-first job dispatcher for Go, designed to bring peace of mind to your background jobs.

The Hidden Dangers of "Just a Goroutine"

When you run background tasks in production, you face a minefield of potential failures:

  • Silent Panics: A panic inside a goroutine can take down the whole service if not recovered. If recovered, the job simply stops, half-finished, with no record of its failure.
  • Retry Storms: If a dependency is down, thousands of jobs retrying at once can create a "thundering herd" problem, making the outage even worse.
  • Lost Jobs on Restart: Any jobs waiting in a simple channel-based queue will vanish into thin air the moment your application restarts.
  • Duplicate Execution: During a failover, you might accidentally run the same job twice, leading to inconsistent data or duplicate emails sent to users.

CSJD is designed from the ground up to tackle these exact problems.

Introducing CSJD: Your Reliability Co-pilot

CSJD is a lightweight, dependency-free library that wraps your job handlers with production-grade safety features. It supports two modes:

  1. In-Process: Perfect for single-service applications with low operational overhead.
  2. Distributed: Uses Redis Streams for a robust, multi-node consumer group setup.

Here’s how CSJD protects your jobs:

  • Persistence: It saves jobs to a file or Redis, ensuring they survive restarts.
  • Smart Retries: Failed jobs are automatically retried with exponential backoff and jitter, preventing retry storms.
  • Panic Safety: It recovers from panics within your handlers, logs the error, and safely moves the job to a dead-letter queue.
  • Controlled Concurrency: You define a fixed worker pool, so you never have unbounded goroutines overwhelming your system.
  • Dead-Letter Queue: After too many retries, or if a job fails permanently, it's moved to a DLQ for later inspection. No job is ever truly lost.

Show Me The Code!

Let's see how easy it is to use. Here's a quick example of the in-process dispatcher.

First, install it:

Now, let's set up a dispatcher that saves its state to a local file.

package main import ( "context" "fmt" "time" "github.com/puretype-spec/CSJD/dispatcher"
) func main() { // Jobs will be persisted to this file store, _ := dispatcher.NewFileJobStore("./data/jobs.json") defer store.Close() // Configure the dispatcher with 4 concurrent workers d, _ := dispatcher.New(dispatcher.Config{ Workers: 4, MaxPendingJobs: 10000, Store: store, }) // Register a handler for "send_email" jobs _ = d.RegisterHandler("send_email", func(ctx context.Context, job dispatcher.Job) error { fmt.Printf("Processing job %s: Sending email...
", job.ID) // Your email sending logic here... // If this returns an error, CSJD will retry it! // If it panics, CSJD will catch it! return nil }) // Start the dispatcher in the background _ = d.Start() fmt.Println("Dispatcher started.") // Submit a new job! job := dispatcher.Job{ID: "job-123", Type: "send_email"} _ = d.Submit(job) fmt.Printf("Submitted job %s.
", job.ID) // Gracefully stop the dispatcher on shutdown fmt.Println("Waiting 5 seconds before shutdown...") stopCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second) defer cancel() _ = d.Stop(stopCtx) fmt.Println("Dispatcher stopped gracefully.")
}
